# Story — Feature Test Coverage Matrix

> Auditable coverage plan for Story. This document enumerates the
> user-visible features that must be tested before Story can claim
> "every feature is covered". It is a test specification for humans
> and workers; it does not implement tests.

## Gate Policy

Story has two kinds of gates:

| Gate | Cadence | Purpose |
|---|---|---|
| `npm run test:cljs` | default CI | Fast CLJ/CLJS contract tests for registry, runtime, pure UI projections, and testbed integration. |
| `npm run test:examples` | default CI when browser examples are in scope | Browser smoke across runnable examples and Story shell surfaces already hosted by examples. |
| `npm run test:story-static` / `npm run story:build` | default CI for static export changes | Static-build contract and deployable output sanity. |
| `npm run test:story-feature-load` | occasional / pre-commit only until proven stable | Full browser feature-load and resilience gate. This is intentionally not required on every CI run yet; it should graduate only after flake rate, runtime, and diagnostic quality are acceptable. PR #1198 adds the first candidate gate but is not assumed merged by this spec. |
| `node ../tools/story/bench/bundle-size.cjs` | release / size-sensitive changes | Story-on, Story-off, and static bundle size comparison. |

The 20-event re-check below means: run a deterministic burst of 20
user-visible events against the currently selected Story shell, then
re-assert the same feature contract. The burst should use low-level
domain events that produce trace epochs, action rows, and scrubber
history without changing the feature under test. For counter surfaces,
prefer a repeatable `inc` / `dec` sequence that lands back on the
original count. For login surfaces, prefer deterministic submit /
failure / retry / dismiss loops with network effects stubbed.

The 20-event gate is an occasional / pre-commit stress check, not a
default every-CI requirement until it proves stable.

## Coverage Status Vocabulary

| Status | Meaning |
|---|---|
| Covered | Current `main` has a direct automated check for the contract and a meaningful failure mode. |
| Partial | Current `main` checks some of the contract, or PR #1198 adds likely browser coverage, but at least one required path, diagnostic, or 20-event re-check is still missing. |
| Deferred (bd:rf2-xxxxx) | A precise follow-up bead owns the missing path; the row is intentionally not Covered until that bead lands. The bead id is recorded in the row's Status cell. |
| Missing | No direct automated check exists yet. |

A row is **never** Covered "because a different probe runs on the same
shell". The feature-load gate (`tools/story/test/story_feature_load.cjs`)
makes this an explicit, code-level commitment via its `COVERAGE_MATRIX`
table: every row is either `kind: 'probe'` (the gate exercises the
named feature) or `kind: 'owned-by'` (the gate declares the row
out-of-scope and names the gate that does cover it). Re-using one probe
for two rows is allowed only when the probe genuinely exercises both
features it is registered against; using a tangentially-related probe
to claim coverage — the aliasing pattern this section warns about — is
forbidden. The header comment on `COVERAGE_MATRIX` is the authoritative
rule.

## Required Testbed Affordances

The existing testbeds stay the baseline:

| Testbed | Must expose |
|---|---|
| `tools/story/testbeds/counter_with_stories` | Small deterministic state, four core variants, toolbar modes, faceted tags, custom decorator, custom panel, workspaces, force-fx-stub, controls, actions, trace, scrubber, a11y root stamping, static export. |
| `tools/story/testbeds/login_form` | Machine states, async/fx-shaped login flow, force-fx-stub, five variants, light/dark mode axis, authenticated/error/submitting states. |

Additional deliberately interesting surfaces should be added by a
follow-up implementation bead, not here:

| Affordance | Shape |
|---|---|
| Handler exception variant | Event handler throws during lifecycle or play; shell records and displays the error without killing the page. |
| Render failure variant | Registered view throws during render; canvas and diagnostics panel show variant id, view id, source, and stack. |
| Loader success / timeout / rejection variants | Loaders complete normally, never satisfy `:loaders-complete-when`, and reject/throw. |
| Failing assertion variant | At least two failing `:rf.assert/*` rows plus one passing row prove record-don't-throw aggregation. |
| Async/fx failure variant | Stubbed managed effect returns failure and emits a trace/action/error record. |
| Schema/args validation variant | Controls produce invalid args and schema-validation panel reports exact path, expected schema, and actual value. |
| Frame isolation pair | Two variants dispatch the same events in adjacent frames and prove app-db, emitted fx, trace history, scrubber history, and assertions do not leak. |
| Trace/scrubber cascade variant | At least six cascaded events/effects produce multiple epochs so scrubber selection, trace filtering, and release can be checked. |

## PR #1198 Alignment

PR #1198 (`test:story-feature-load`) is the right home for the first
wide browser gate. When merged, it should be checked against this
matrix and extended rather than treated as complete coverage. In

## Diagnostic Minimums

Every browser failure in the full feature-load gate should include:

| Diagnostic | Required content |
|---|---|
| Feature id | Matrix row name or stable slug. |
| Route and selection | URL, selected variant/workspace, active tab, active modes, active filters. |
| Frame context | Frame id, variant id, parent story id, current app-db snapshot or redacted summary. |
| Trace context | Last cascade id, last epoch id, scrubbed/live state, recent action rows. |
| Assertion context | Assertion ids, pass/fail counts, expected/actual/source for failures. |
| Error context | Error kind, phase, event/fx/loader/decorator/view id, source coord, stack. |
| Screenshot/DOM | Screenshot on browser failure plus relevant `data-test` selector counts. |
| Network context | Unexpected requests, failed requests, and opt-in state for allowed egress. |

If a gate cannot provide these diagnostics, its status should remain
Partial even if it exercises the happy path.
